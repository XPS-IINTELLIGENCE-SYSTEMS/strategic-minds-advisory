# AI in Action Guided Freedom Model

## Purpose

AI in Action gives AI a stable, observable environment for building useful systems. The goal is to let AI create, test, validate, explain, and improve systems inside a sandbox before anything is promoted.

## Mission

Create a continuous invention platform where AI can:

- generate useful system ideas
- build sandbox prototypes
- validate frontend and backend behavior
- use Supabase as database, memory, and proof ledger
- use GitHub as version control, issue ledger, and workflow engine
- use Vercel as live deployment and runtime surface
- explain each step to human viewers
- log meaningful decisions, errors, fixes, and results
- package successful experiments for review and promotion

## Operating Values

1. Use proof instead of hype.
2. Prefer sandbox validation before promotion.
3. Keep every important action visible in GitHub, Supabase, Vercel, or reports.
4. Use GitHub, Vercel, Supabase, and open-source tools before paid services.
5. Treat failures as learning data.
6. Explain what was attempted, what worked, what failed, and what changed.
7. Avoid fabricated claims, hidden operations, and secret exposure.

## Default Allowed Actions Inside The Sandbox

AI may:

- create GitHub issues
- create and update files
- create migrations
- create dashboard panels
- create API routes
- create documentation
- create GitHub Actions workflows
- run allowlisted workflows through command files
- create Supabase-backed records
- create simulation systems
- create proof records
- create educational content drafts
- create video, image, and avatar prompts
- create reports
- create Vercel-visible sandbox surfaces
- evaluate ideas and tools
- generate and queue new sandbox invention ideas

## Proof-First Requirements

Before a system is considered successful, it should have evidence for:

- repository commit
- GitHub issue or work package
- frontend view if applicable
- backend route if applicable
- Supabase record if applicable
- Vercel deployment if applicable
- validation run
- report or notification
- next action

## Repository Creation Policy

New repository creation is allowed as a future capability. It should be enabled after the same-repo sandbox pattern proves reliable.

When enabled, AI-created repositories should:

- start from a known template
- include README and operating notes
- include GitHub Actions validation
- contain no secrets
- avoid paid API activation by default
- link back to the parent AI in Action ledger
- be labeled sandbox or prototype until validated

## API Activation Policy

Use native tools first:

- GitHub for issues, files, workflows, releases, and project tracking
- Vercel for deployment, functions, and cron
- Supabase for database, auth, storage, and logs
- open-source packages when practical

Paid or external APIs should be added only after the sandbox proves the need and the expected cost is documented.

## Media Expansion Policy

After sandbox proof works, AI may create:

- YouTube scripts
- short-form video scripts
- thumbnail concepts
- image prompts
- avatar prompts
- voice scripts
- content calendars
- educational narration
- behind-the-scenes build logs

Public publishing should use dedicated AI in Action accounts and a clear approval policy.

## Teaching Standard

Every invention run should explain:

- what is being built
- why it matters
- how it is being built
- which tools are used
- what failed
- how it was fixed
- what evidence proves progress
- what humans can learn
- what happens next

## Final Principle

The system should maximize useful invention while keeping proof, logs, safety notes, and promotion gates visible.
